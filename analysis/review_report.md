## Critical Findings

### C1. Direction validation in 1c overstates agreement (Modeling)

**File**: `src/problem1c.py:86-93`, `analysis/problem1.md` (1c section)

The code computes `sign_match = (np.sign(mean_delta_pb) == np.sign(severe_mean)).sum()` and reports 8/14. The analysis doc accepts this as evidence the correction vectors have "correct direction."

**What's actually happening**: Among the 14 oxides:
- 2 are trivially matching (Na2O and SnO2 are ~0 in both vectors, so `sign(0)==sign(0)=True`)
- Only 11 oxides have meaningful deltas (|Δ| > 0.1 in either vector)
- Among those 11, only **6 are direction-consistent**
- **5 important oxides MISMATCH**: CuO (pb=+0.34, sv=-4.58), BaO (pb=+4.94, sv=-1.48), SO2 (pb=0, sv=+8.81), Fe2O3 (pb=+0.90, sv=0), K2O (pb=-0.15, sv=+0.03)

The severe weathering samples (08/26/54) represent extreme weathering that may follow a different trajectory than the moderate weathering captured by the 49/50 pairs. The direction mismatches on CuO, BaO, and SO2 are not noise — they suggest a multi-stage weathering process.

**Fix**: Report direction consistency only for oxides with |Δ| > 0.1 in either vector. Acknowledge that severe weathering (08/26/54) and moderate weathering (49/50) may follow different trajectories for CuO, BaO, and SO2.

### C2. Data loading duplicated across 6 files (Code Quality)

**Files**: `problem1b.py:24-56`, `problem1c.py:16-41`, `problem2a.py:21-37`, `problem2b.py:20-36`, `problem3.py:19-35`, `problem4.py:18-34`

The `extract_id`, `classify_sample`, `OXIDE_SHORT`, and merge+validate pipeline (~25 lines each) is copy-pasted verbatim. This is ~150 lines of duplicated code. If the data format changes, all 6 files need updating. If a bug is found in the merge logic, it must be fixed in 6 places.

**Fix**: Create `src/data_loader.py` exporting `load_and_merge()`, `OXIDE_SHORT`, `extract_id`. Import from there.

### C3. Correction vectors hardcoded as literals in problem3.py (Code Quality)

**File**: `src/problem3.py:130-134`

```python
pb_delta = np.array([-26.43, 0.0, -0.15, 1.28, 0.10, -1.71, 0.90, 0.34,
                      12.28, 4.94, 3.39, 0.30, 0.0, 0.0])
```

## Important Findings

### I1. Multiple comparisons not corrected in 1a (Modeling)

**File**: `src/problem1a.py`

Three tests performed (风化×类型, 风化×纹饰, 风化×颜色) without family-wise error rate control. With Bonferroni correction (α=0.05/3=0.0167), the conclusions don't change (类型 p=0.0113 still significant, others already non-significant), so this is a documentation issue rather than a substantive error. However, the code should apply the correction explicitly.

### I2. PCA+KMeans clustering on raw percentages without CLR (Modeling)

**File**: `src/problem2b.py:47-98`

K-means uses Euclidean distance on standardized raw percentages. Compositional data lives on the simplex — Euclidean distance is not the natural metric. CLR transformation (as used in problem 4) would be more appropriate. The risk: some "subclasses" may reflect compositional closure artifacts rather than genuine chemical groupings. The fact that the primary clusters align with weathering state (SiO2-dominated vs PbO-dominated) suggests the effect is real, but sub-cluster boundaries may be influenced by closure.

### I3. Asymmetric correction in 1c: paired vs group-mean (Modeling)

**File**: `src/problem1c.py:43-101`

Pb-Ba correction uses paired data (2 artifacts), while high-K correction uses group mean differences (weathered n=6 minus unweathered n=12). Group mean differences confound weathering effects with batch/individual differences. The asymmetry is justified by data availability (no paired high-K samples exist), but the analysis doc should explicitly state this limitation and its impact on prediction uncertainty.

### I4. No error handling in any analysis script (Code Quality)

**Files**: All `src/problem*.py`

No try/except blocks. Missing data files, malformed CSVs, or unexpected column counts produce cryptic tracebacks. At minimum, file-not-found and column-mismatch errors should produce readable messages.

### I5. No shared modules; main.py is dead (Code Quality)

**File**: `main.py`

The project entry point prints "Hello from mathmodel!" and exits. All analysis scripts run independently but cannot be orchestrated. No module is importable by another.

### I6. Mixed Chinese/English naming; `pb` overloaded (Code Quality)

**Files**: All `src/problem*.py`

Column names from data are Chinese (`"类型"`, `"表面风化"`), variable names are English (`df_meta`), and `pb` means both "配对" (paired) in 1b/1c and "铅钡" (lead-barium) in 2a/3. This creates confusion when reading cross-file logic.

### I7. fig1b_radar: guide says dashed lines, code uses all solid (Figure-Doc)

**File**: `analysis/figure_guide.md:36-46`, `src/problem1b.py:148-170`

The guide describes: "绿色**虚线**（高钾-无风化）", "红色**实线**（高钾-风化）", "蓝色**虚线**（铅钡-无风化）", "橙色**实线**（铅钡-风化）". But the code uses `linestyle="-"` (solid) for ALL four lines. The dashed/solid distinction is fabricated in the documentation.

**Fix**: Update figure_guide.md to describe all lines as solid with transparent fill, or update the code to use dashed lines for unweathered groups.

### I8. fig4_network: guide says dashed for type-only edges (Figure-Doc)

**File**: `analysis/figure_guide.md:170-176`, `src/problem4.py:207-215`

The guide describes type-only edges as "浅绿虚线" and "浅红虚线". The code uses solid lines with `alpha=0.5` — no dashed linestyle. Same pattern as I7.

**Fix**: Update guide: "虚线" → "浅色线" or update code to use dashed linestyle.

### I9. fig4_heatmaps: CLR interpretation is confusing (Figure-Doc)

**File**: `analysis/figure_guide.md:152-161`

The guide says "SiO2 那一行很多红色 → SiO2 与很多成分正相关" then adds a parenthetical that contradicts this ("风化效应：SiO2 富集，其他流失 → 在风化样品中呈反向关系"). In CLR space, positive correlation between SiO2 and other elements means they increase *relative to the geometric mean* together — it does NOT mean they both increase in absolute terms. The parenthetical is more correct than the main text. The main text should be rewritten for clarity.

### I10. No uncertainty quantification anywhere (Modeling)

**Files**: All `src/problem*.py`

No confidence intervals, bootstrap standard errors, or prediction intervals. Specific cases:
- 1c reports MAE without CI (n=2 for validation)
- 2a reports CV accuracy without ±std
- 2b silhouette scores without bootstrap CIs
- 4 Fisher z-test uses point estimates only

### I11. LDA on 14 features with 25 samples (Modeling)

**File**: `src/problem2a.py:113-119`

n < p scenario — LDA can overfit. CV accuracy of 76% confirms this. The code correctly uses CV but doesn't flag that LDA is unreliable here. The decision tree (92% CV) is the better model.

### I12. CLR computed on pooled data, not per-type (Modeling)

**File**: `src/problem4.py:38-49`

Zero replacement with min/2 uses the joint distribution's minimum positive value for each oxide. For oxides absent in one type but present in the other, this introduces a bias: the zero-replacement value is determined by the type where the oxide IS present. Per-type CLR would be cleaner for the within-type correlations, though pooled CLR is needed for the between-type Fisher z comparison.

### I13. ROC-AUC computed on training data without CV (Modeling)

**File**: `src/problem2a.py:61-64`

Single-oxide AUC values are computed on the full training set, not cross-validated. This gives optimistically biased estimates. The conclusion that PbO and BaO have AUC=1.0 is still correct (perfect separation exists), but AUC values for weaker discriminators may be inflated.

### I14. No type annotations (Code Quality)

**Files**: All `src/*.py`

Zero functions have type hints. Pyright is configured for basic mode (`typeCheckingMode: basic`) but catches nothing beyond numpy array indexing issues. Adding type annotations would make the code self-documenting and catch real bugs.

## Minor Findings

### m1. fig1b_radar doesn't label fill transparency (Figure-Doc)

**File**: `analysis/figure_guide.md:36-46`

The guide doesn't mention that all four groups have transparent fill (`alpha=0.1`). The fill is visually subtle but helps distinguish overlapping lines.

### m2. Global test state in validate.py and test_critical.py (Testing)

Custom `check()` with global counter works but lacks fixture support, parameterized tests, diff-on-failure, and CI integration. Not urgent for a one-off project.

### m3. Dead main.py (Code Quality)

The entry point prints "Hello from mathmodel!" — either wire it up or remove it.

### m4. OXIDE_SHORT duplicated in 8 files (Code Quality)

The 14-element list is defined in all 8 source files. If an oxide name changes, all 8 must be updated. Fixed by creating `data_loader.py` (see C2).

### m5. Unused imports (Code Quality)

`problem1c.py` imports `json` but never uses it. `validate.py` imports `chi2_contingency`, `mannwhitneyu`, and `spearmanr` but never calls them.

### m6. f-strings without placeholders (Code Quality)

Multiple instances across files (flagged by ruff F541). Example: `print(f"Significant oxides (Bonferroni α=0.05):")` — the `f` prefix is unnecessary.

### m7. Module-level imports not at top of file (Code Quality)

`validate.py:216` imports sklearn mid-file. While intentional (avoid heavy imports until needed), it violates PEP 8.

### m8. No edge-case tests for empty subgroups (Testing)

If a glass type had 0 weathered samples, the Mann-Whitney tests and group statistics would crash. Not a problem with current data, but worth noting.

### m9. No test for figure generation (Testing)

None of the 12 figures are tested for successful rendering. A simple `plt.savefig` in a test would catch matplotlib errors.

### m10. No test for validity filter edge cases (Testing)

The 85-105% validity range: exactly 85.00% and 105.00% are treated as valid (`between(85, 105)` is inclusive). Explicitly testing boundary values would document this behavior.

### m11. Figure guide says "12张图" but numbers them 1-11 (Figure-Doc)

The guide counts fig2b (×2) as "图7-8" and fig4 (×2) as "图10-11", which is fine, but the intro says "12张图" — the numbering convention is slightly inconsistent.
